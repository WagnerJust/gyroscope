## Next
- [ ] **Make gyroscope-maintained repos first-class for devs who don't have gyroscope installed.** Principle: adopting gyroscope must not tax contributors who never installed it — *using* the repo needs nothing, and *contributing* needs nothing mandatory. Runtime is already clean: the SessionStart hook is pure `cat … 2>/dev/null || true` (no binary), and the hub/spokes/pointers + the `.claude/agents/` persona mirror are plain markdown any agent or human reads natively — a tool-less dev's sessions already work. The friction is in **maintenance + legibility**, not usage:
  - **CI gating.** If an adopting repo wires `gyroscope check` into CI, an outside contributor's PR can fail a gate they can't run or understand. Options: keep the check advisory / non-blocking for external PRs, and/or document a zero-install invocation — gyroscope is `go install`-clean (cobra-only), so `go run github.com/WagnerJust/gyroscope/cmd/gyroscope@latest check .` should need no prior install (verify + document).
  - **Hand-edits must degrade gracefully.** A dev who edits the hub's managed region, adds a spoke, or edits a persona *without* running gyroscope must never hard-break anything — the next `check --fix` (by someone who has it, or CI) reconciles. Confirm every managed artifact is reconcilable-after-the-fact and not order-dependent. The `<!-- gyroscope:managed -->` markers already fence "don't hand-edit inside"; make that discoverable.
  - **Legibility ("what is this?").** A stranger sees `gyroscope.json`, `<!-- gyroscope:managed -->` markers, `.local/`, and injected hub text that says "run `/gyroscope`" — meaningless without the tool. Add a short contributor-facing explainer (a `## For contributors` note in the hub, or a routed spoke / CONTRIBUTING.md line): "these docs are gyroscope-maintained; you don't need it to read or contribute; here's what the markers mean; running it is optional (`go run …@latest`)." Make the `/gyroscope` + `gyroscope.json` references degrade to something a tool-less reader understands.
  - **Open decisions (scope before building):** (a) explainer home — hub managed region vs separate spoke vs CONTRIBUTING.md; (b) whether `init` should emit the outside-contributor note automatically; (c) how the recommended CI wiring should treat forks / external PRs.
- [ ] **Skill: adopt-reconcile step for pre-existing personas (`skill/SKILL.md` persona flow).** The persona flow is greenfield-only (ask → pick templates → write → mirror); it has no branch for a repo that already has personas. `gyroscope check` is *config-relative* — a `skipped`/`unknown`/`off` state means it never inspects `docs/agents/` and green says nothing about persona files on disk. So adopting into a repo with existing personas (esp. in a `docs/agents/personas/` subdir the root-only mirror can't see) reads as "conformant, nothing to do" when personas are silently unregistered. Add a **first** step to the persona flow: scan `docs/agents/**` for `name:`-bearing files regardless of config state, classify root-canonical (ADR 0010) vs subdir (doc-routed-on-purpose OR mislaid — **ask**, don't assume), reconcile state-vs-disk disagreement, then fall through to wire/skip. **Binary is not touched** — it stays config-relative and deterministic; judgment lives in the skill (the binary/skill split). Motivated by the exodus adopt session; see the doc-routed-persona pattern in `review-checkpoints.md` and ADR 0010.
